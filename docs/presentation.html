Motivation

Why is Indic OCR Hard?

Traditional OCR like Tesseract works like a photocopier that reads letter-by-letter. Indic scripts don't work that way — and they break it completely.

❌ Why Traditional OCR Fails

  • Tamil/Telugu letters fuse together visually — one "letter" on screen is actually 3–4 Unicode symbols combined. Tesseract slices right through the middle and reads garbage.
  • Traditional OCR converts images to black & white before reading. The tiny hairline strokes that distinguish vowel marks in Tamil get erased in this step.
  • Newspaper pages have 2–3 columns. Without layout awareness, the text comes out interleaved — col-A line 1, col-B line 1, col-A line 2... completely unreadable.
  • No understanding of what is body text vs page header/footer — all noise gets included.

✅ How AI Solves This

  • Our Vision Transformer looks at the entire line as a picture and directly types out the characters — like a skilled human typist, not a letter-by-letter scanner.
  • We never convert to black & white. The model trained on full-colour images and relies on subtle greys to distinguish vowel marks.
  • A geometric algorithm (X-Y Cut) traces the invisible column boundaries and reads left column top-to-bottom first, then right column — exactly as a human would.
  • An AI layout classifier labels every region (Body, Header, Footer, Image) so we only extract actual content.

INPUT

Any scanned document: PDF (up to 100 pages), PNG/JPG/TIFF image, or a Word DOCX file with embedded images.

PROCESSING

An AI Vision model on a cloud GPU reads every text line. It takes roughly 2–5 seconds per page on an NVIDIA T4 GPU.

OUTPUT

A clean, downloadable .txt file with the text in the correct reading order, ready to search, copy, or feed into other tools.

Cost trick: The website (UI) runs always-on on a cheap CPU server. The expensive GPU only switches on when you click "Run OCR" — and switches off the moment it’s done. Zero idle cost.

Stage 1

Document Ingestion — Converting to Images

Before any AI touches the document, we convert every page into a raw pixel image. Think of it as "photographing" each page so the model can see it clearly — just like a human would look at a physical document.

📄 PDF Files

We open the PDF and handle two cases: if a page is just a scanned photo embedded inside, we pull the raw image out directly (no quality loss). If the PDF has mixed text and graphics, we re-render the page at 300 DPI — high enough to keep even the finest vowel mark hairlines sharp.

⚠️ Max: 100 pages per run

🖼️ Image Files

Supports PNG, JPG, TIFF, BMP, WEBP. We always convert to RGB colour mode. This is critical — some PNGs have a transparent alpha channel, some GIFs use indexed colours. The AI model expects plain 3-channel colour, and any mismatch would cause a silent crash.

📝 DOCX Files

Word documents often have scanned pages pasted as images inside them. We walk through the file's internal links to find and extract each embedded image blob, then treat each one as a separate page. Up to 100 images supported.

KEY DECISION — WHY WE NEVER CONVERT TO BLACK & WHITE

Old OCR systems convert pages to black & white to "clean them up". We deliberately don't. The thin grey hairlines that form vowel marks in Tamil and Telugu are too faint — they disappear the moment you binarize, and the model misreads or completely skips those characters. Surya was trained on full-colour images, so we keep them that way.

Stage 2

The AI OCR Engine — How the Model Reads

Two cooperating AI models work together — think of them as a Highlighter and a Typist.

🔦 Model 1 — The Highlighter (Detection)

This model scans the entire page image and draws a box around every text line it finds — one box per sentence or paragraph fragment. It doesn't read any words yet. It just marks where text is.

Output: coordinates for each box + a confidence score saying how sure it is

✍️ Model 2 — The Typist (Recognition)

Each highlighted line crop is handed to this model. It looks at the image of that line and types out the characters, one by one, until the full line is written. It has no fixed list of allowed letters — it can produce any Unicode character in any script.

No fixed vocabulary — works on any script it was trained on

💡 Why ViT, Not the Old Way?

  • Old systems read letter by letter, slicing the image into thin columns. Tamil letters fuse across multiple columns — slicing right through them produces garbage.
  • The Vision Transformer looks at the whole line at once as a single image patch. Compound characters are seen as a whole, not split apart.
  • Its attention mechanism naturally picks up long-range dependencies — e.g. a vowel mark on the right that changes the meaning of the consonant on the left.

📊 What Each Line Returns

For every detected line, the engine gives back: the text it read, the bounding box (where on the page), and a confidence score from 0 to 1.

Scores below 0.90 are flagged for the NLP cleaning stage. Scores below 0.30 are discarded entirely as unreadable noise.

Stage 3

The Layout Engine — Reading the Page Like a Human

After the Typist writes out the text, we need to understand what kind of text each region is. A raw list of lines has no idea what's a headline, what's a footnote, and what's an image caption.

🏠 The Document Organizer

The same AI backbone that powered the OCR engine now looks at the full page as a whole and labels every region with its semantic type. Because it reuses the same model weights, no extra GPU memory is consumed. It recognises 14 region types:

TextSectionHeaderPageHeaderPageFooter CaptionListItemTablePicture FootnoteFigureEquationCode

🧹 The Cleanup Crew (IoU Deduplication)

The AI often draws multiple overlapping boxes on the same paragraph — like stacking highlighters on the same sentence. We measure how much each pair of boxes overlaps (this ratio is called IoU). If two boxes overlap more than 70%, we keep only the more confident one and throw the other away. This collapses 40+ duplicate detections into one clean block.

🚫 What Gets Filtered Out

  • Images, figures, tables, equations — there is no text to extract from these regions, so we skip OCR on them entirely.
  • Page headers & footers — these are stripped out (book title repeating on every page, page number, etc.) unless they contain actual body text.
  • Page numbers — detected by regex: pure digits, Roman numerals, and Tamil "பக்கம்" patterns are all caught.

🔄 Safety Net — Fallback Mode

Sometimes the AI misreads an entire newspaper page as one big "Picture" and finds no text blocks at all. When this happens, the engine creates a single full-page block covering the entire image and runs OCR on it anyway. Nothing gets skipped — we always produce output.

Stage 4

Fixing Reading Order — The X-Y Cut Algorithm

After layout detection we have labelled blocks scattered across the page. But in what order should they be read? A newspaper has 3 columns. A book has 1. A magazine mixes both. This algorithm figures it out — with no AI, just geometry.

Step 1 — Look for a Vertical Split

Imagine drawing a vertical line down the page. If you can find a position where no text block crosses that line, the page has two columns. We read the entire left column first, top-to-bottom, then repeat for the right column. Then we split each column the same way — recursively.

Step 2 — Look for a Horizontal Split

If no vertical split exists, draw a horizontal line. A full-width headline above a 2-column body creates a natural horizontal split. We read the headline first (top band), then recurse into the body below it.

Step 3 — Just Sort Top-to-Bottom

If the page has no splits at all — like a normal single-column storybook — we simply sort all blocks from top to bottom, then left-to-right for anything at the same height. Done.

📐 Why This Matters

Without this, a newspaper page would produce completely scrambled output — alternating lines from different columns:

Line 1 col-A   Line 1 col-B
Line 2 col-A   Line 2 col-B
Line 3 col-A   Line 3 col-B

With X-Y Cut, it becomes perfectly readable:

Line 1 col-A
Line 2 col-A
Line 3 col-A
Line 1 col-B
Line 2 col-B
Line 3 col-B

Stage 5

Cleaning Up the Text — NLP Refinement

The Typist model is very good, but not perfect. The final stage runs a deterministic cleaning pass — no AI, just rules — to fix known artifacts before the text reaches you.

🧹 Noise Removal

Vision models sometimes generate junk characters that look like text but aren't. We use regex patterns to strip them out:

  • Stray HTML tags: <br>, </p>
  • Pipe characters used as fake column dividers: |
  • Long underscore runs: ____
  • Repeated ellipsis: .....
  • Empty bracket pairs: [ ], { }

🔤 Unicode Fix (NFC)

Indic scripts have a hidden problem: the same letter can be stored two different ways in memory. For example, the Tamil letter "கி" can be one combined character OR a consonant followed by a separate vowel mark — they look identical on screen but are different byte sequences.

OCR engines often produce the split form. We apply NFC normalisation to merge them, so search, copy-paste, and text comparison all work reliably.

📝 Line Reflow

Narrow layout blocks sometimes produce very short orphan lines — a few words dangling alone. Left as-is, these become excessive line breaks in the final output that confuse downstream tools.

The reflow pass merges consecutive short lines (under 18 characters) back into a single paragraph, producing clean, readable prose.

METRICS COMPUTATION

If the user provides a ground-truth reference, we compute:

  • CER (Character Error Rate) — edit distance per character
  • WER (Word Error Rate) — edit distance per word token

Before comparison, both texts are stripped of page headers, page numbers, structural artifacts, and normalized to a single space-separated string.

OUTPUT FORMAT

Deployment

Two Servers, One Smart Trick

GPUs are expensive. The key insight: you only need the GPU for a few seconds per document. So we split the system into two parts — a cheap always-on web server, and a powerful GPU that only wakes up when needed.

🖥️ CPU Server — Always On

  • Serves the website and upload form to users 24/7
  • No GPU — runs on a cheap CPU-only instance that costs almost nothing
  • Accepts your file, shows the progress log, and delivers the download link
  • Think of it as the reception desk — always there, no heavy lifting
@app.function(image=cpu_image, min_containers=0)

⚡ GPU Server — On Demand Only

  • Completely switched off until you click "Run OCR"
  • Boots up, loads the AI models (~15 seconds cold start), processes your document
  • Streams live progress updates back to the UI as it works
  • The moment it finishes — it shuts down. You pay for those seconds only.
@app.function(gpu="T4", min_containers=0)

📡 How They Talk to Each Other

When you click Run OCR, here's exactly what happens step by step:

  1. The CPU server reads your uploaded file into memory as raw bytes
  2. It sends those bytes to the GPU server and says "start processing"
  3. The GPU server sends back live status messages one by one as it works — that's the streaming log you see
  4. When done, it sends a final "done" message and the CPU server gives you the download link

The live log you watch in the browser is powered by this real-time message stream between the two servers.

Future Scope

Next Step — Teaching the System to Self-Correct

Even at 93% accuracy, ~7% of characters are wrong. The next planned upgrade adds an AI proofreader that reads the output and fixes the remaining mistakes. Not implemented yet — planned for Phase 3.

🤔 Why Can't a Normal AI Fix This?

  • ChatGPT-style models break text into chunks of common word-pieces before reading. A single OCR typo in Tamil (one wrong character) destroys the entire chunk — the model can't recognise the word at all.
  • ByT5 reads raw bytes instead — every single character is its own token. A one-character typo is just one bad byte among many good ones, and the model can easily fix it in context.
  • This makes ByT5 uniquely suited for character-level surgical correction in complex Indic scripts.

🎯 The Correction Strategy

  • Only lines with a confidence score below 0.90 are sent to ByT5 — high-confidence lines are left untouched so we don't slow things down unnecessarily.
  • Lines with known junk patterns (HTML tags, pipe characters, underscore runs) are also corrected regardless of their score.
  • The corrector runs as a completely separate step after the main pipeline — it can be switched off without touching anything else.

📈 Expected Improvement

  • Target: reduce error rate from ~7% → ~4% on clean modern print
  • Bigger gains on degraded archival documents where the OCR model struggles most
  • Uses google/byt5-small — a compact model that fits alongside Surya on the same T4 GPU
