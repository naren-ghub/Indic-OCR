Motivation

Why is Indic OCR Hard?

Traditional OCR systems like Tesseract were built for Latin scripts. Indic languages break every assumption they make.

❌ The Old Approach Fails Because...

  • Indic scripts use akshara clusters — multiple Unicode codepoints fuse into one visual unit
  • Tesseract binarizes images, destroying vowel mark hairlines
  • Column layout interleaving produces garbled reading order
  • No built-in way to separate body text from headers/footers
  • Completely fails on degraded archival prints (≥40% CER)

✅ What We Do Instead

  • Use a Vision Transformer that reads the whole line as an image — no fixed vocabulary
  • Keep images in full RGB colour — no binarization
  • Apply Recursive X-Y Cut to reconstruct correct reading order geometrically
  • Detect and prune structural blocks (PageHeader, PageFooter) before assembly
  • Apply Unicode NFC normalization to fix invisible encoding bugs
Tesseract CER: ~40% on archives Surya CER: ~7% on modern print 11 Indic Languages Supported

Stage 1

Document Ingestion & Pre-processing

The pipeline accepts three document types and converts every page to a high-fidelity RGB PIL Image before any AI model touches it.

📄 PDF Files

Opens with fitz.open(). For pages with a single embedded image and no selectable text, we extract the raw bitmap directly with doc.extract_image() — bypassing JPEG recompression. For layered PDFs, we rasterize at 300 DPI for maximum detail.

⚠️ Max: 100 pages per run

🖼️ Image Files

Supports PNG, JPG, JPEG, TIFF, BMP, WEBP. Loaded directly with Pillow. Always converted to RGB — this removes alpha channels (RGBA PNGs) and palettized modes (P-mode GIFs) that would crash the ViT encoder.

📝 DOCX Files

Iterates over doc.part.rels to extract all embedded images by relationship type. Each image blob is opened with Pillow and converted to RGB. Up to 100 images are processed as individual pages.

KEY DESIGN DECISION — NO BINARIZATION

Traditional OCR pipelines binarize (threshold to black/white) images to reduce noise. We deliberately skip this step. Surya's Vision Transformer is trained on full-colour RGB images. Binarization destroys the subtle grey-level information in vowel mark hairlines (especially in Tamil & Telugu), causing the model to misread or drop characters entirely.

Stage 2

Surya Vision Transformer OCR Engine

The recognition core uses two cooperating transformer models — one for geometry, one for reading.

🔲 Model 1 — Line Detection

A CNN/Transformer hybrid scans the full-page image and outputs line-level bounding boxes in pixel coordinates. Each box tightly frames one text line — a single sentence or part of a paragraph.

Output: list of [x0, y0, x1, y1] + per-line confidence scores

📖 Model 2 — Sequence Recognition

Each detected line crop is fed into an encoder-decoder ViT. The encoder maps the image patch to a dense feature vector. The decoder autoregressively generates the Unicode character sequence, one token at a time.

No fixed character vocabulary — works on any script it was trained on

💡 Why ViT Over CRNN?

  • CRNNs slice images column-by-column — Indic akshara clusters span multiple columns and get split incorrectly
  • ViT patches cover the entire receptive field simultaneously — compound characters are seen as a whole
  • Attention mechanism naturally handles right-to-left reordering of vowel marks

📊 Output Format

Each line returns a dict: {"text": "...", "bbox": [...], "confidence": 0.97}

Confidence scores below 0.90 are flagged for surgical correction in the NLP stage. Lines below 0.30 are discarded entirely.

Stage 3

Cognitive Layout Analysis Engine

Raw OCR lines are just a bag of text — they have no structure. The Layout Engine understands the page as a semantic document.

🏷️ Surya LayoutPredictor

Runs on the same FoundationPredictor backbone as the OCR engine — no second GPU load. Classifies every region of the page into one of 14 semantic labels:

TextSectionHeaderPageHeaderPageFooter CaptionListItemTablePicture FootnoteFigureEquationCode

🔍 Block Deduplication (IoU)

The model often returns multiple overlapping predictions for the same region. We compute Intersection-over-Union (IoU) between all bounding box pairs. If IoU > 0.70, the lower-confidence block is dropped. This collapses 40+ duplicate boxes into a clean single block.

🚫 Semantic Filtering

  • SKIP_LABELS: Picture, Figure, Table, Equation — these blocks contain no extractable text. OCR is not run on them.
  • STRUCTURAL_LABELS: PageHeader, PageFooter — stripped unless they contain real body text (detected via regex)
  • Page number detection: Regex matches pure digits, Roman numerals, and Tamil "பக்கம்" patterns

🔄 Fallback Logic

If the model returns no text-class blocks (e.g., a dense newspaper page misclassified entirely as Picture), the engine creates a single full-page fallback block covering 100% of the image. This ensures OCR always runs, even on unusual layouts.

Stage 4

Recursive X-Y Cut Reading Order

The most important innovation in the pipeline. Reconstructs the human-intended reading order from geometry alone — zero GPU cost, 100% deterministic.

Step 1 — Find a Vertical Cut

Project all block bounding boxes onto the X-axis. Search for a column gutter: an X coordinate where no block's left edge < X < right edge. If found within the middle 70% of the page width, this is a valid two-column split. Recurse into left column and right column separately.

Step 2 — Find a Horizontal Cut

If no vertical gutter exists, look for a horizontal band: a Y coordinate where no block straddles it. This handles a full-width headline above a two-column body. Process top band first, then recurse into the bottom.

Step 3 — Y-Sort Fallback

If neither cut is found (single-column storybook), fall back to a simple top-to-bottom sort by Y coordinate — then by X for ties. This is the path taken for the majority of Tamil books in our test set.

📐 Why This Matters

Without layout-aware reading order, a two-column article would produce:

Line 1 col-A   Line 1 col-B
Line 2 col-A   Line 2 col-B
Line 3 col-A   Line 3 col-B

With X-Y Cut it correctly becomes:

Line 1 col-A
Line 2 col-A
Line 3 col-A
Line 1 col-B
Line 2 col-B
Line 3 col-B
Handles single-column Handles two-column Handles mixed layouts
Stage 5

NLP Refinement & Unicode Normalization

Post-recognition, the assembled text passes through a deterministic NLP cleaning layer before output.

🧹 Noise Suppression

Regex patterns strip well-known OCR hallucination artifacts:

  • Stray HTML tags: <br>, </p>
  • Pipe characters used as column separators: |
  • Long underscore runs: ____
  • Repeated ellipsis: .....
  • Empty bracket pairs: [ ], { }

🔤 Unicode NFC Normalization

Indic scripts encode compound characters in multiple valid ways. The Tamil letter "கி" can be represented as one precomposed codepoint OR as two separate codepoints (consonant + vowel mark). OCR engines often produce the decomposed form. We apply unicodedata.normalize("NFC", text) to ensure all downstream search, diff, and comparison tools work correctly.

📜 Text Reflow

The layout engine sometimes yields very short "orphan" lines (fewer than 18 chars) from narrow block crops. The reflow pass merges consecutive short lines into a unified paragraph, preventing excessive line breaks in the final output that would confuse downstream NLP models.

METRICS COMPUTATION

If the user provides a ground-truth reference, we compute:

  • CER (Character Error Rate) — edit distance per character
  • WER (Word Error Rate) — edit distance per word token

Before comparison, both texts are stripped of page headers, page numbers, structural artifacts, and normalized to a single space-separated string.

OUTPUT FORMAT

Deployment

CPU + GPU Microservice Architecture

The biggest engineering decision: splitting the app into two independent Modal containers to eliminate idle GPU billing.

🖥️ CPU Container — Always On

  • Serves the Gradio web UI and FastAPI endpoint
  • No GPU — runs on standard CPU-only instance
  • Accepts file uploads, renders the interface, streams logs
  • Costs fractions of a cent per hour — negligible billing
@app.function(image=cpu_image, min_containers=0)

⚡ GPU Container — On Demand Only

  • Cold-starts only when user clicks "Run OCR"
  • Provisions NVIDIA T4 GPU, loads Surya models (~15s)
  • Processes document, yields JSON progress events via remote_gen()
  • Writes output .txt to shared Modal Volume, then scales to zero
@app.function(gpu="T4", min_containers=0)

📡 Inter-Container Communication

When the user clicks Run OCR, the CPU container:

  1. Reads the uploaded file into raw bytes in memory
  2. Calls run_ocr_gpu.remote_gen(file_bytes, langs, ...)
  3. Iterates the generator — each event is a JSON string
  4. Parses event type: progress → update log, done → set download

The live streaming log the user sees in the UI is powered by this JSON event stream flowing from the GPU container back to the CPU container in real time.

Future Scope

Neural Language Correction — ByT5

The next planned upgrade: a character-level Language Model pass that fixes residual OCR errors the Vision model couldn't catch. Not implemented in the current deployment.

🔬 Why ByT5?

  • Standard LLMs use subword tokenizers — a single OCR typo fragments a Tamil word into unrecognizable pieces
  • ByT5 operates on raw UTF-8 bytes — every character is its own token, with no vocabulary mismatch possible
  • Uniquely suited to surgically fix character-level OCR garbling in complex Indic scripts

🎯 Correction Strategy

  • Only route lines with confidence < 0.90 through ByT5 — high-confidence lines are left untouched for throughput
  • Lines with known noise patterns (HTML tags, pipes, underscore runs) also get corrected regardless of score
  • Runs as an isolated post-processing step — does not alter the core OCR pipeline

📈 Expected Impact

  • Target CER improvement: ~7% → ~4% on clean modern print
  • Stronger gains expected on degraded archival prints where Surya leaves more residual errors
  • Model: google/byt5-small — compact enough for T4 inference alongside Surya
